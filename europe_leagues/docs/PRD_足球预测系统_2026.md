## 1. 背景与目标

`europe_leagues` 已从“若干脚本并列存在”的形态，收敛为一条正式主链：

- 采集：`collect-data`
- 预测：`predict-match` / `predict-schedule`
- 持久化：`domain/persistence.py`
- 结果闭环：`save-result` / `auto-sync-results` / `result-sync-daemon`
- 复盘与治理：`sync-pending-results-review` / `build-season-master-review`
- 运行维护：`health-check` / `setup-openclaw` / `refresh-repo-docs`

本 PRD 的目标不是重复执行手册，而是明确：

- 当前系统的正式边界
- 哪些能力属于主链
- SoT 与 runtime-only 的分工
- 哪些说明应以 CLI-first 方式理解

## 6. 端到端正式流程

### 6.1 赛前流程

1. `collect-data` 获取赛程、`match_id`、上下文与可复用快照信息
2. `predict-match` / `predict-schedule` 执行预测
3. 通过 `domain/persistence.py` 统一处理 SoT / runtime-only 写回、archive、MEMORY、RAG、registry side effects
4. 如需阶段化审计，使用 `harness-run`

### 6.2 赛后流程

1. `pending-results` 查看待回填比赛
2. `auto-sync-results` 尝试自动同步
3. 未命中时使用 `save-result` 手工补录
4. 如需批次回填 + 复盘，总入口为 `sync-pending-results-review`
5. 如需显式重建统计，再执行 `accuracy --refresh`

### 6.3 自动闭环事实

正常赛果闭环后，系统会推动多项衍生产物刷新，包括但不限于：

- SoT 比分 / 备注
- `MEMORY.md`
- `prediction_archive.json`
- 准确率统计
- RAG 记忆样本与索引
- review-learning 相关输出

因此 `accuracy --refresh` 是显式重建入口，不应被描述成唯一常规路径。

## 9. 当前风险与治理重点

当前最容易产生漂移的区域不是模型本身，而是“说明口径”：

- 文档把兼容入口写成主逻辑
- 文档和 skill 引用了失效路径
- README / guide / PRD / skills 对 SoT 与 runtime-only 的描述不一致
- 开发态 Python 调试示例被误当成正式使用方式

治理原则：

- 入口说明以 `app/cli.py` 为准
- 结果闭环说明以 `domain/persistence.py`、`runtime/result_sync.py`、`result_manager.py` 为准
- 技能说明与文档统一采用 CLI-first 叙述
- repo-local 路径必须真实存在，否则删除引用

---

## 10. 验收标准

文档与 skill 收口完成后，应满足：

1. 仓库内不再出现“正式入口是谁”的多套说法
2. 文档列出的正式命令都真实存在于 `app/cli.py`
3. 文档中的 repo-local 路径都真实可达
4. README / guide / PRD / skills 对 SoT/runtime-only 边界说法一致
5. 默认示例以 CLI 为主，底层 Python import 只保留给调试说明
