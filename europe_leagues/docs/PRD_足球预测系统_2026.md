## 1. 背景与目标

### 1.1 背景
本项目 `europe_leagues` 是面向五大联赛（英超/西甲/德甲/意甲/法甲）的比赛预测与分析系统，核心能力包括：
- 赛程维护（按轮次更新，避免误导比分）
- 球队/球员数据维护（中文名、号码、统计、高阶数据、热区等）
- 赔率落盘与未来赔率快照（用于相似盘路与爆冷分析）
- 预测生成与报告落盘（Markdown 报告）
- 赛果回填与准确率统计（按联赛、按模型）

### 1.2 目标（可量化）
- 能稳定生成未来 N 天五大联赛预测报告（默认 3 天，可扩展）。
- `players/*.json` 不再出现“全队空/球队输入退化导致预测内容一致”的问题。
- 赛程段落更新后：未完赛比分统一为 `-`（避免 0-0 误导）。
- 将“球员更新”“赛程更新”封装为可复用 Skill，后续一键统一更新。

## 2. 产品范围（In/Out）

### 2.1 In Scope（本 PRD 覆盖）
- 五大联赛赛程更新：写回 `teams_2025-26.md` 的赛程段落。
- 五大联赛球员数据更新：名单 → 中文名/号码 → 统计/热区 → 审计。
- 预测系统运行链路：输入数据读取优先级、报告生成、缓存、结果统计。
- 文件管理规范：目录、产物、缓存、模板、脚本职责。

### 2.2 Out of Scope（暂不承诺）
- 全部联赛官方站点的完整伤病/首发阵容/训练报道等深度数据抓取（站点反爬强，需额外投入）。
- “评分/热图（触球热区）/传球成功率/防守对抗”等来自 WhoScored 的全量字段（抓取与授权成本高）。
  - 当前落盘：以 Understat/FPL/Sofascore/Wikidata 可稳定获取字段为准，并标注来源。

## 3. 用户与使用场景

### 3.1 目标用户
- 预测分析使用者：需要每天生成预测、关注爆冷风险、复盘准确率。
- 数据维护者：需要定期更新赛程、球员数据、赔率快照。

### 3.2 典型场景
- 场景 A：预测前，先统一更新赛程和球员数据，保证输入可信。
- 场景 B：转会窗后，一键刷新全量球员名单、号码、统计和热区。
- 场景 C：发现某联赛赛程/球员数据异常，快速重刷并输出审计报告定位缺失。

## 4. 系统总体架构（高层）

### 4.1 核心模块
| 模块 | 入口/文件 | 说明 |
|---|---|---|
| 统一入口 | `prediction_system.py` | 菜单式启动增强预测、结果录入、统计查看 |
| 增强预测 | `enhanced_prediction_workflow.py` | 主预测链路（动态调权、相似盘路、爆冷分析、报告生成） |
| 旧流程兼容 | `optimized_prediction_workflow.py` | 优先调用增强预测器，失败回退旧逻辑 |
| 模型融合 | `ml_prediction_models.py` | 多模型融合输出（支持动态权重） |
| 结果管理 | `result_manager.py` | 赛果录入、准确率统计、按模型统计 |

### 4.2 数据分层（输入优先级）
- 赛程：`teams_2025-26.md`（赛程段落为权威落盘）
- 球员：`<league>/players/*.json`
- 赔率：
  1) `analysis/odds_snapshots/`（未来赛程即时快照）
  2) `analysis/odds/`（历史赔率落盘）
  3) 缺失时降级（报告会提示）

## 5. 端到端流程（确保可运行）

### 5.1 推荐执行顺序（每日/每周）
1. 更新赛程（五大联赛）
2. 更新球员数据（五大联赛）
3. 生成预测报告（未来 N 天）
4. 赛后录入结果并更新准确率

### 5.2 预测生成
- 入口：`python3 prediction_system.py`（推荐）
- 输出：各联赛 `analysis/predictions/` 下 `YYYY-MM-DD_predictions*.md`

### 5.3 缓存与可追溯
- `.prediction_cache/`：增强预测的 TTL 缓存（24h），可删（仅影响性能，不影响正确性）。

### 5.4 人工增强赛前分析流程（新增标准）
当用户要求对单场比赛做深度赛前预测时，统一采用下面这条链路，避免“只看模型”或“只看单一外部文章”：

1. 读取本地基础数据：
   - `teams_2025-26.md` 的赛程、积分、近期结果
   - `players/*.json` 的进球、助攻、xG、xA、关键传球与可出场信息
2. 读取爆冷知识库：
   - 默认接入 `爆冷案例库.json`
   - 有命中时作为风险修正项；无命中时明确写明“无直接案例支持”
3. 抓取临场市场数据：
   - 欧赔、亚盘、可稳定复核的凯利
   - 至少做 2 个外部来源交叉，不使用无法复核的单点截图结论
4. 做冲突校验：
   - 若本地模型、赔率方向、外部排名互相冲突，必须显式说明采用哪套口径
   - 不允许把来源冲突的数据直接混写成同一个结论
5. 输出统一结论：
   - 胜平负主结论
   - 2 至 3 个建议比分
   - 大小球倾向
   - 爆冷等级与是否需要防平/防冷
6. 结果落盘：
   - 赛前结论写入对应联赛 `teams_2025-26.md` 的预测记录区
   - 赛后再补实际结果、预测正确与复盘结论

## 6. 模块拆解与文件管理（强规范）

### 6.1 联赛目录规范（五大联赛一致）
| 路径 | 作用 |
|---|---|
| `<league>/teams_2025-26.md` | 球队信息 + 赛程信息 + 积分榜 + 数据来源 |
| `<league>/players/*.json` | 球队球员数据（名单/中文名/号码/统计/热区/来源） |
| `<league>/analysis/predictions/` | 预测报告输出与模板 |
| `<league>/analysis/results/` | 赛果模板与结果落盘（如有） |
| `<league>/analysis/odds/` | 历史赔率落盘 |
| `<league>/analysis/odds_snapshots/` | 未来赔率快照落盘 |

### 6.2 输出文件命名规范
- 预测报告：`YYYY-MM-DD_predictions.md` / `YYYY-MM-DD_predictions_enhanced.md`
- 赔率快照：`*_odds_snapshot.json`
- 缓存文件：`.prediction_cache/<md5>.json`（内部实现细节）

## 7. 数据规范（字段/来源/追溯）

### 7.1 players/*.json（核心字段规范）
| 字段 | 类型 | 说明 |
|---|---|---|
| `name` | string | 展示用中文名（优先），无法确认则保留英文 |
| `english_name` | string | 英文名（对齐 Understat/Sofascore/FPL） |
| `wikidata_id` | string | Wikidata QID（用于中文名/号码追溯） |
| `sofascore_player_id` | number | Sofascore ID（用于阵容/号码补齐追溯） |
| `shirt_number` | int/string | 单一号码（若唯一） |
| `shirt_numbers` | array | 多个号码备选（Wikidata 可能给出历史号码） |
| `stats.*` | number | 出场、分钟、进球、助攻、黄红牌等 |
| `advanced_stats.*` | number | Understat xG/xA 等高阶统计 |
| `heatmap.*` | object | 基于 Understat 射门坐标聚合的热区网格 |
| `transfer_status` | string | `current/injured/suspended/...`（英超可稳定补齐） |
| `injury_reason` | string | 伤病/缺阵原因（英超来自 FPL news） |
| `data_sources` | array | 标注来源，如 `understat.com/fantasy.premierleague.com/sofascore.com` |

### 7.2 teams_2025-26.md（赛程表格规范）
统一表头（所有联赛保持一致）：
| 日期 | 时间 | 主队 | 比分 | 客队 | 备注 |
|---|---|---|---|---|---|

强制规则：
- `status != 已结束` 时：`比分 = '-'`
- `备注` 用于标注：`已结束/进行中/待验证` + 业务补充

### 7.3 赛前预测记录规范（新增）
各联赛的赛前人工分析，需要统一补写到各自 `teams_2025-26.md` 的“预测记录”区，建议格式如下：

- 表格字段：
  - 日期
  - 比赛
  - 预测结果
  - 预测比分
  - 实际结果
  - 实际比分
  - 预测正确
- 未赛前的记录要求：
  - `实际结果 = -`
  - `实际比分 = -`
  - `预测正确 = 待赛后验证`
- 表格下方补 2 至 4 条“赛前分析摘要”，至少说明：
  - 本次采用的核心口径（模型/赔率/人工复核）
  - 关键支撑因素（战意、盘口、球员状态、爆冷案例）
  - 若出现模型与市场相反，必须写“人工覆盖模型”原因

### 7.4 多源数据验证边界（新增）
为避免赛前报告出现伪精确或冲突结论，统一约束如下：

- 本地文件优先级：
  - 赛程与落盘状态以 `teams_2025-26.md` 为主
  - 球员产出与高阶数据以 `players/*.json` 为主
- 外部信息用途：
  - 赔率、盘口、凯利、伤病快讯、媒体前瞻用于补强
  - 外部排名若与本地积分表冲突，不直接覆盖本地表
- 冲突处理规则：
  - 若外部文章排名和本地积分冲突，优先比较“盘口方向 + 本地球员产出 + 本地近期赛果”
  - 若模型输出与市场明显相反，不直接采用模型结论，需人工复核后再落盘
  - 若凯利数据无法稳定复核，可以不报具体数值，但必须说明原因
- 明确不承诺项：
  - 不凭空生成首发名单
  - 不在缺少稳定来源时伪造凯利数值
  - 不把单一媒体文案当作最终结论

## 9. 运行保障与验收标准

### 9.1 运行保障（最低要求）
- 任意一天：能完成赛程更新与预测报告生成，无人工编辑依赖。
- `players/*.json` 不应出现全空导致“球队强度退化为默认值”的情况（若某队为空，审计应能定位）。

### 9.2 验收指标（建议）
- 赛程更新后：随机抽查 10 场未完赛，比分均为 `-`
- 球员数据：
  - 五大联赛每队至少 18 名球员（低于 18 的队触发“名单可能不全”提示）
  - 中文名命中率持续提升（允许保守跳过歧义短名）
  - 球衣号码缺失率可接受（来源不同可分层验收）

### 9.3 单场深度预测验收清单（新增）
一场纳入“正式赛前分析记录”的比赛，至少满足以下检查项：

- 已读取本地赛程和积分信息
- 已读取两队 `players/*.json`
- 已查询 `爆冷案例库.json`
- 已获取至少 1 组可复核欧赔/亚盘数据
- 若引用凯利，必须来自可稳定复核页面
- 若存在数据源冲突，已在结论里明确采用口径
- 已写入联赛 `teams_2025-26.md` 的预测记录区

## 10. 风险与待优化点（优先级）

### P0（必须）
- 赛季参数可配置（Understat/FPL/官网文章 URL 可能每赛季变化），建议增加 `--season` 参数与统一配置文件。
- 跨源球员对齐：建议引入稳定 ID（Wikidata QID / Sofascore ID / FPL ID）做主键，降低误匹配风险。

### P1（重要）
- 中文名歧义处理：输出“待人工确认清单”，支持人工 CSV 覆盖（例如 `Manuel Ángel`、`Raúl`）。
- 数据来源可追溯增强：报告头部写入 players/odds/schedule 的更新时间戳与来源摘要。

### P2（增强体验）
- 评分/传球/防守等细项：若后续引入授权数据源（WhoScored/Opta等），可扩展字段，但必须标注来源与版本。
